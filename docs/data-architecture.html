GEDCOM Data Architecture - Upload, Query, and Delete Flows

Slide 2

Where Data Lives

Three storage layers, each with a distinct role

R2  Object Storage

  • Raw .ged files - one per user or anon session
  • Demo trees: demo/golden-rosenberg.json
  • Geocoding: geocodes/gazetteer.json
  • Anon: expires in 1 day. Users: 7 days from last login
  • Cleaned by nightly cron at 02:00 UTC

D1  SQLite (Cloudflare)

  • sessions - anon session IDs and expiry
  • users - Clerk user ID, email, encrypted API key, GEDCOM expiry
  • demo_individuals, demo_events, demo_families, demo_sources - pre-seeded, queryable by Claude (planned)

Browser  Client-Side State

  • Parsed GEDCOM in JS memory (individuals, events, families)
  • Map visualization state (particles, filters, clusters)
  • localStorage.kf_returning - suppresses welcome on return visits
  • localStorage.kf_hidden_demos - which demo trees are hidden
  • localStorage.CHAT_KEY_LS - Anthropic key for regular users

Local  Chat Proxy (Dev Only)

  • Python server at localhost:8787
  • SQLite DB populated from browser-parsed GEDCOM
  • Claude executes SQL tool calls against this DB in dev
  • Not present in production - Cloudflare functions handle Claude there
Slide 3

User Tiers

Three tiers determine GEDCOM retention and Claude access

Anonymous

Session cookie only. Can view demo trees and upload a GEDCOM (1-day retention). No Claude access in production. No auto-restore on return.

Regular

Clerk login. GEDCOM stored 7 days from last login. Claude requires own Anthropic API key stored in localStorage. Tree auto-restores on sign-in.

VIP

Clerk login + email on VIP allowlist. Full Claude access using the app's Anthropic API key. Same 7-day GEDCOM retention as Regular.

Slide 4

Upload Flow

User saves their GEDCOM to the cloud via "Save to Account"

User opens a .ged file
1
User → Browser  Selects or drops a .ged file
2
Browser  Parses GEDCOM + geocodes events client-side using the gazetteer
3
Browser  Renders map visualization; sets localStorage.kf_returning = "1"
User saves to account
4
User → Browser  Clicks "Save to Account"
5
Browser → Middleware  POST /api/gedcom/upload with Authorization: Bearer {token} and raw .ged body
6
Middleware  Verifies Clerk JWT; extracts user_id and email; upserts users row in D1
7
Middleware → R2  put("gedcom/user/{user_id}", body)
8
Middleware → D1  SET gedcom_expires_at = now + 7 days
9
Middleware → Browser  Returns { stored: true, expires_at: unix_ts }
10
Browser → User  "Saved. Expires {date}."

Key Points

  • Raw .ged text is stored - not parsed data
  • One file per user - upload overwrites previous save
  • Retention: 7 days for auth, 1 day for anon
  • JWT verified on every request; no server-side session

Auto-Restore on Sign-In

  • Clerk resolves on page load
  • If kf_returning flag is set, skip welcome screen
  • App calls GET /api/gedcom with Bearer token
  • Fetches raw .ged from R2 and parses in browser
  • "Restoring your tree..." shown in chat during fetch
Slide 5

Query Flow - Claude Chat

Three paths depending on user tier and environment

Dev - Local Proxy
1
User types a question in chat
2
Browser → Proxy  POST /v1/messages (Anthropic format)
3
Proxy  Runs SQL tool calls against local SQLite DB (populated from the GEDCOM)
4
Proxy → Anthropic  Sends messages + tool results
5
Anthropic → Proxy → Browser  SSE stream
6
Browser  Renders answer in chat panel
VIP - Production
1
User types a question in chat
2
Browser → CF /api/claude/chat  POST with Authorization: Bearer {token}
3
CF  Verifies JWT; confirms VIP tier; uses app ANTHROPIC_API_KEY
4
CF → Anthropic  Messages API call
5
Anthropic → CF → Browser  SSE stream
6
Browser  Renders answer in chat panel
Regular - Own Key
1
User types a question in chat
2
Browser  Reads Anthropic key from localStorage
3
Browser → Anthropic  POST /v1/messages directly, x-api-key: {key}
4
Anthropic → Browser  SSE stream (no Cloudflare hop)
5
Browser  Renders answer in chat panel
!
Key never leaves the browser - not sent to Cloudflare
Slide 6

Delete Flow

User trees deleted from server. Demo trees hidden locally only.

User tree - permanent server delete
1
User → Browser  Clicks X on their own tree in the TREES panel
2
Browser → Middleware  DELETE /api/gedcom with Authorization: Bearer {token}
3
Middleware  Verifies JWT; resolves user_id
4
Middleware → R2  delete("gedcom/user/{user_id}")
5
Middleware → D1  SET gedcom_expires_at = NULL
6
Middleware → Browser  204 No Content
7
Browser  Removes tree from local state and map visualization
Demo tree - local hide only (no server call)
1
User → Browser  Clicks hide (eye icon) on a demo tree
2
Browser  Adds tree name to localStorage.kf_hidden_demos
3
Browser  Removes tree from visualization - no server request made
↻
User can restore via "Restore demos" - clears the flag and reloads from /api/demo
Nightly cron - automatic expiry (02:00 UTC)
★
Delete from R2 + D1 where anon sessions.expires_at < now
★
Delete from R2 where users.gedcom_expires_at < now. Demo trees never touched.

User Tree Delete

  • Removes .ged from R2
  • Nulls expiry in D1
  • Clears browser state
  • Irreversible - must re-upload
  • Only available to auth users

Demo Tree Hide

  • Client-side only (localStorage)
  • Server data never deleted
  • Reversible via "Restore demos"
  • Per-browser - not synced across devices
Slide 7

Demo Tree Flow

Shared trees available to all users - read-only on server

App startup (any user tier)
1
Browser → /api/demo  GET (no auth required)
2
CF → R2  get("demo/golden-rosenberg.json")
3
R2 → CF → Browser  Pre-processed JSON (individuals + events + families)
4
Browser  Loads tree into map visualization (no client-side parsing needed)
User hides a demo tree
5
User → Browser  Clicks hide on demo tree
6
Browser  Adds name to localStorage.kf_hidden_demos; removes from visualization
User restores hidden demo trees
7
User  Clicks "Restore demos"
8
Browser  Clears kf_hidden_demos; calls GET /api/demo again
9
Browser  Re-adds demo tree to visualization

Demo Data Sources

  • R2: Pre-processed JSON - fast, no parse step
  • D1: demo_* tables - seeded for future Claude SQL queries
  • Seeded once via scripts/seed-demo.mjs
  • Never modified or deleted by any user action

Planned: Gregory-Henry Tree

  • Second demo at demo/gregory-henry.json
  • GET /api/demo/list returns all demo names
  • Both trees load on startup unless hidden
  • Each independently hideable per-browser
